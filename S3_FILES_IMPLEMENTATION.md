# Amazon S3 Files 導入ガイド

本ドキュメントでは、Amazon S3 Files を使用して、Amazon S3 バケットを AWS Lambda のファイルシステムとしてマウントし、SQLite データベースの永続化を実現するための実装方法をまとめます。

## 1. 概要と導入のメリット

Amazon S3 Files は、S3 バケットを NFS プロトコル経由でファイルシステムとしてアクセス可能にするサービスです。

### 導入前の方式（従来設計）
- Lambda のコールドスタート時に S3 から SQLite ファイルを `/tmp` にダウンロード。
- データ更新ごとに S3 へアップロード（または Lambda 終了時にアップロード）。
- **課題**: 同期ロジックの実装が複雑、同時実行時の整合性確保が困難、S3 API の呼び出しコスト。

### 導入後の方式（S3 Files）
- S3 バケットを Lambda のディレクトリ（例: `/mnt/s3`）に直接マウント。
- アプリケーションは標準的なファイル I/O（SQLite）でそのまま読み書き。
- **メリット**:
    - **実装の簡素化**: S3 同期用のコードが不要になり、標準の SQLite 操作のみで完結。
    - **低レイテンシー**: アクティブなデータは高性能ストレージ層にキャッシュされ、ミリ秒単位のアクセスが可能。
    - **整合性の向上**: NFS の整合性モデルにより、複数インスタンスからのアクセス管理が容易。

## 2. インフラ設定手順

### 2.1 S3 ファイルシステムの作成
1. **S3 コンソール**を開き、「**File systems**」を選択します。
2. 「**Create file system**」をクリックします。
3. マウント対象とする **S3 バケット**を選択します。
4. ファイルシステム名を入力し、作成を完了します。

### 2.2 VPC とマウントターゲットの設定
S3 Files は VPC 内のネットワークエンドポイントを経由してアクセスします。
1. **マウントターゲットの作成**: 各アベイラビリティゾーンのサブネットに対してマウントターゲットを作成します。
2. **セキュリティグループ**: NFS（ポート 2049）の通信を許可するセキュリティグループを設定し、Lambda 関数とマウントターゲットに適用します。

### 2.3 アクセスポイントの設定（推奨）
特定のパスへのアクセス制限や、POSIX ユーザー/グループ ID（UID/GID）の強制を行うために、アクセスポイントを作成します。

---

## 3. Lambda 関数の設定

### 3.1 VPC 設定
Lambda 関数を、S3 Files のマウントターゲットが存在する VPC およびサブネットに接続します。

### 3.2 ファイルシステムのマウント
1. Lambda 関数の「**設定**」タブから「**ファイルシステム**」を選択します。
2. 「**ファイルシステムを追加**」をクリックします。
3. 作成した S3 ファイルシステムとアクセスポイントを選択します。
4. **ローカルマウントパス**を指定します（例: `/mnt/s3`）。

### 3.3 IAM 権限の追加
Lambda 関数の実行ロールに以下の権限が必要です：
- `elasticfilesystem:ClientMount`
- `elasticfilesystem:ClientWrite`
- `s3:GetObject`, `s3:PutObject`, `s3:ListBucket`（S3 Files が内部的に使用するため）

---

## 4. アプリケーションコードの変更

### 4.1 データベースパスの変更
環境変数などで SQLite のデータベースパスをマウントポイント配下に設定します。

```go
// 例: internal/config/config.go
func GetDBPath() string {
    path := os.Getenv("DB_PATH")
    if path == "" {
        // デフォルトはマウントパス配下を指すようにする
        return "/mnt/s3/app.db"
    }
    return path
}
```

### 4.2 手動同期ロジックの削除
これまで検討・実装されていた、起動時の `DownloadFromS3` や更新時の `UploadToS3` といった処理をすべて削除します。

---

## 5. 注意事項

- **コスト**: S3 Files のストレージ料金（高性能ストレージ層に配置されたデータ分）と、S3 本体のストレージ料金、および同期リクエスト料金が発生します。
- **整合性**: NFS v4.1/4.2 の close-to-open 整合性に基づきます。SQLite のような同時書き込みが発生する用途では、引き続き Lambda の同時実行数を制限する、あるいは SQLite の WAL モードを適切に設定するなどの検討が必要です。
- **初回アクセス**: 初回の読み込み時は S3 からデータがインポートされるため、レイテンシーが発生する可能性があります。
