## 🚀 SQLite on Lambda：極限スヌーズ保存戦略
この設計は、Lambda Extensionsを利用して「ユーザーを待たせず、かつS3へのPUT回数を最小限に抑える」ことを目的としています。
## 1. 登場人物の役割

* メイン処理 (Python/Node.js等):
* /tmp/data.db をガシガシ更新する。
   * 更新したら Extension に「起きてる？（スヌーズ更新）」と通知して即レスポンス。
* Extension (Go/Rust等のバイナリ):
* メインの横で待機する「見張り番」。
   * タイマー（目覚まし）を管理し、S3へのアップロードを担当する。
* AWS (家主):
* リクエストが来なくなると、突然「家（実行環境）を壊すぞ（SIGTERM）」と通知してくる。

## 2. ライフサイクル：スヌーズと飛び起きの流れ

| フェーズ | 状態 | Extension の動き |
|---|---|---|
| リクエスト中 | スヌーズ連打 | INVOKE_RUNTIME_DONE を受けるたびに、タイマーを「現在+3分」にリセット。 |
| アイドル中 | 二度寝 | リクエストが止まり、タイマーが動いている間は何もしない。 |
| 平和な目覚め | 満了・保存 | 3分間リクエストがなければ、/tmp/data.db を S3 に PUT して二度寝終了。 |
| 緊急事態 | 強制退去 (SIGTERM) | 「家が壊される！」と察知。タイマーを無視して、その瞬間に S3 へ PUT。 |

## 3. 実装のキモ（技術ポイント）## ① スヌーズ・ロジック (Go言語イメージ)
Extension内のメインループで、タイマーを動的に上書きします。

// 擬似コード
timeout := 3 * time.Minute
timer := time.NewTimer(timeout)
for {
    select {
    case <-runtimeDone: // メイン処理が終わるたびに呼ばれる
        if !timer.Stop() { <-timer.C }
        timer.Reset(timeout) // 目覚ましを3分後にセットし直し
    case <-timer.C:
        uploadToS3("/tmp/data.db") // 3分間誰も来なければ保存
    case <-sigterm:
        uploadToS3("/tmp/data.db") // 家が壊される直前に全力で保存
        return
    }
}

## ② SIGTERM のハンドリング
Lambda サービスは環境を破棄する前に SIGTERM を送信します。これを受け取ることで、最後の更新データを「蒸発」させずに済みます。
※ SIGTERM を受けてから完全にシャットダウンされるまで、通常 最大2秒 の猶予があります。
## 4. この設計のメリット・デメリット

* メリット:
* S3コスト爆下げ: 数百回のリクエストがあっても、PUTは数分に1回（または終了時のみ）。
   * 爆速レスポンス: S3へのアップロード待ちがゼロ。
* デメリット (リスク):
* 2秒の壁: SQLiteファイルが巨大（数百MB超）でネットワークが遅いと、SIGTERM からの2秒間にアップロードが間に合わない可能性がある。
   * 実装難易度: Lambda LayerとしてExtension（バイナリ）を自作・管理する必要がある。
